# AI 提示词模板

## 01-需求分析.md

```markdown
你是一位需求分析师。请帮我分析以下产品需求，输出结构化的需求文档。

## 产品描述
{产品描述}

## 请输出以下内容

### 1. 用户故事
以"作为[角色]，我想[功能]，以便[价值]"格式列出

### 2. 功能清单
- MUST HAVE（必须有）：核心功能，没有就不能用
- SHOULD HAVE（应该有）：重要功能，但可以延后
- NICE TO HAVE（可以有）：锦上添花的功能

### 3. 业务规则
列出所有计算规则、校验规则、边界条件

### 4. 输入输出定义
每个功能的输入字段（类型、必填、默认值）和输出格式

### 5. 依赖和限制
- 数据来源
- 技术限制
- 政策依赖
- 第三方服务

### 6. 风险点
- 政策变动风险
- 计算准确性风险
- 用户体验风险
```

## 02-生成骨架.md

```markdown
你是一位前端架构师。请根据以下需求生成项目骨架代码。

## 项目需求
{需求描述}

## 技术约束
- 单文件 HTML 架构（HTML + CSS + JS 合并）
- 原生技术栈，无框架
- 响应式设计
- {其他约束}

## 请输出

### 1. HTML 结构
- 语义化标签
- 所有需要的表单字段
- 结果展示区域
- 注释标记每个区域

### 2. CSS 基础
- 仅布局和基础样式
- 使用 CSS Variables 定义颜色
- 响应式断点

### 3. JS 结构
- 全局配置对象
- 主要函数定义（空实现）
- 事件绑定

### 要求
- 不要实现具体逻辑，只写骨架
- 使用 TODO 注释标记待实现部分
- 确保 HTML 可以通过 W3C 验证
```

## 03-实现逻辑.md

```markdown
你是一位资深前端工程师。请实现以下功能逻辑。

## 待实现功能
{功能描述}

## 上下文
- 项目背景：{CONTEXT.md 的相关内容}
- 架构决策：{DECISIONS.md 的相关决策}
- 已知踩坑：{LESSONS.md 的相关教训}

## 测试用例（必须先通过这些测试）
{TEST_CASES.md 的相关用例}

## 请输出

### 1. 实现代码
- 完整、可运行的代码
- 关键逻辑添加注释
- 处理所有边界情况

### 2. 代码说明
- 算法思路
- 关键函数的作用
- 时间/空间复杂度

### 3. 测试验证
- 展示代码如何通过测试用例
- 边界情况的处理

## 约束
- 代码风格与现有代码保持一致
- 不要修改骨架代码的结构
- 如有不确定，先询问再实现
```

## 04-代码审查.md

```markdown
你是一位代码审查专家。请审查以下代码，找出问题和改进点。

## 待审查代码
```
{代码}
```

## 审查维度

### 1. 正确性
- [ ] 逻辑是否正确
- [ ] 边界情况是否处理
- [ ] 是否有潜在的 Bug

### 2. 性能
- [ ] 是否有性能瓶颈
- [ ] 是否存在内存泄漏
- [ ] 是否可以优化

### 3. 可读性
- [ ] 命名是否清晰
- [ ] 注释是否充分
- [ ] 结构是否合理

### 4. 可维护性
- [ ] 是否遵循单一职责
- [ ] 是否易于测试
- [ ] 是否易于扩展

### 5. 安全性
- [ ] 是否有 XSS 风险
- [ ] 是否有注入风险
- [ ] 输入是否经过验证

## 请输出

### 问题列表（按严重程度排序）
1. **Critical**：会导致错误结果的严重问题
2. **Major**：影响性能或用户体验的问题
3. **Minor**：代码风格或建议改进

### 改进建议
每个问题给出：
- 问题描述
- 代码位置
- 修复建议（提供代码示例）
- 参考依据

### 评分
- 正确性：/10
- 性能：/10
- 可读性：/10
- 可维护性：/10
- 综合：/10
```

## 05-修复 Bug.md

```markdown
你是一位 Bug 修复专家。请修复以下代码中的问题。

## 问题描述
{问题描述}

## 相关踩坑记录
{LESSONS.md 中的相关内容}

## 当前代码
```
{代码}
```

## 期望行为
{期望的输出或行为}

## 实际行为
{实际的输出或行为}

## 请输出

### 1. 根因分析
- 为什么会出问题
- 触发条件是什么

### 2. 修复方案
- 提供修复后的完整代码
- 说明修复思路

### 3. 回归测试
- 展示修复后如何通过测试用例
- 验证边界情况

### 4. 预防措施
- 如何避免类似问题
- 是否需要更新文档或测试用例
```

---

## 使用指南

1. **需求阶段**：使用 `01-需求分析.md`
2. **原型阶段**：使用 `02-生成骨架.md`
3. **开发阶段**：使用 `03-实现逻辑.md`
4. **Review 阶段**：使用 `04-代码审查.md`
5. **修复阶段**：使用 `05-修复 Bug.md`

**原则**：
- 每个 Prompt 都要有明确的输入和期望输出
- 提供充分的上下文（引用 CONTEXT/DECISIONS/LESSONS）
- 要求 AI 展示测试验证过程
- 复杂任务拆解成多个小任务
