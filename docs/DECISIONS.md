# 架构决策记录 (Architecture Decision Records)

## ADR-001: 单文件架构

**状态**: 已接受

**背景**: 需要一个简单、易分享的退休计算器，用户可以直接打开使用，无需配置服务器或安装依赖。

**决策**: 将所有代码（HTML、CSS、JS）合并到一个 HTML 文件中。

**原因**:
- ✅ 用户双击即可打开，零配置
- ✅ 易于通过微信/邮件分享
- ✅ 可部署到任何静态托管（GitHub Pages、Netlify 等）
- ✅ 无网络依赖（除字体外）

**替代方案**:
- ❌ React/Vue 项目：需要构建步骤，过于复杂
- ❌ 多文件分离：增加了分享和部署的复杂度

**后果**:
- 代码组织需要通过注释和区域划分来维护
- 版本控制通过文件名区分（如 v1.html、v2.html）

## ADR-002: 原生技术栈，无框架

**状态**: 已接受

**背景**: 减少外部依赖，提高加载速度，降低长期维护成本。

**决策**: 使用原生 HTML5 + CSS3 + ES6，不引入 React/Vue/jQuery 等框架。

**原因**:
- ✅ 零依赖，加载快
- ✅ 浏览器兼容性更好（尤其是移动端）
- ✅ 代码量小，易于 AI 生成和修改
- ✅ 无构建步骤，开发简单

**替代方案**:
- ❌ React + Webpack：学习曲线陡峭，构建复杂
- ❌ Vue CDN：虽然简单，但增加了 30KB+ 体积

**后果**:
- 需要手动管理 DOM 操作
- 状态管理通过原生 JS 对象实现
- 组件复用通过函数和模板字符串实现

## ADR-003: 渐进式延迟退休计算规则

**状态**: 已接受（基于 2024 年政策草案）

**背景**: 2025 年起实施渐进式延迟退休，需要准确计算不同出生年份的退休年龄。

**决策**: 采用"每 N 个月出生时间延迟 1 个月"的线性插值算法。

**具体规则**:
```
男性：1965-01 至 1976-12，每 4 个月延迟 1 个月
女性干部：1970-01 至 1981-12，每 4 个月延迟 1 个月  
女性工人：1975-01 至 1985-12，每 2 个月延迟 1 个月
```

**算法**:
```javascript
function calculateDelayMonths(birthYear, birthMonth, gender, jobType) {
    const startYear = getStartYear(gender, jobType);  // 1965/1970/1975
    const monthsSinceStart = (birthYear - startYear) * 12 + birthMonth - 1;
    const delayMonths = Math.floor(monthsSinceStart / interval);  // 4 或 2
    return Math.min(delayMonths, maxDelayMonths);  // 36 或 60
}
```

**替代方案**:
- ❌ 查表法：需要维护大表，不够灵活
- ❌ 公式拟合：政策是离散的，线性插值最准确

**后果**:
- 政策如有调整，需要更新算法参数
- 边界月份（如 1965年1月）需要仔细验证

## ADR-004: 省份数据硬编码

**状态**: 已接受

**背景**: 需要各省份的养老金计发基数和缴费上下限。

**决策**: 将数据硬编码在 JS 对象中，不调用外部 API。

**原因**:
- ✅ 数据变化不频繁（每年调整一次）
- ✅ 离线可用
- ✅ 无需处理 API 失败情况
- ✅ 响应即时

**数据格式**:
```javascript
const provinceData = {
    beijing: { base: 11297, upper: 33891, lower: 6782, year: 2024 },
    // ...
};
```

**替代方案**:
- ❌ 调用社保局 API：没有公开的实时 API
- ❌ 本地 JSON 文件：单文件架构限制

**后果**:
- 每年需要手动更新数据
- 新增省份需要修改代码

## ADR-005: 先测试用例，后实现代码

**状态**: 已接受

**背景**: AI 生成代码容易出错，需要验证机制。

**决策**: 采用测试驱动开发（TDD）思路，先写测试用例，再写实现。

**流程**:
1. 人工提供或确认测试用例（输入 → 期望输出）
2. AI 根据测试用例写实现
3. 验证实现是否通过所有测试
4. 如有失败，修复后回归测试

**测试用例格式**:
```markdown
| 用例ID | 输入 | 期望输出 | 状态 |
|--------|------|---------|------|
| TC-001 | 男, 1960-01-01 | 60岁 | ✅ |
| TC-002 | 男, 1970-01-01 | 61岁 | ⏳ |
```

**原因**:
- ✅ 明确预期，减少返工
- ✅ 便于回归测试
- ✅ 文档化边界情况

**后果**:
- 前期需要投入时间设计测试用例
- 需要维护测试用例文档
